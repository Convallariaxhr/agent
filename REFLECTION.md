# REFLECTION.md — Convallaria Coding Agent Harness

## 哪些 Superpowers 技能发挥了最大作用

**冷启动验证（cold-start validation）是最有价值的技能。** 用一个"陌生"agent 来验证 SPEC 和 PLAN 的质量，这个设计非常巧妙。当我自己的 agent 在本地开发环境中"恰好能工作"时，另一个 agent 会在 Windows 路径、GOPROXY 配置、PowerShell 兼容性等问题上立即受阻。这些受阻之处暴露了我未说出口的假设——我最清楚自己的代码，因此最看不到代码中的盲区。冷启动验证发现的 10 个问题中，有 3 个（并发安全、ID 生成 bug、反馈闭环触发时机）是我自己 review 永远发现不了的。

**requesting-code-review 也发挥了重要作用。** 一个独立的 general-purpose agent 审查了我的全部 3400 行代码，找到了 4 个 Critical 问题和 5 个 Important 问题。其中最让我震惊的是"Agent 竞态条件"——我把 `messages` 作为 struct 字段，但 Agent 实例在 HTTP 服务器中被所有请求共享。这是典型的"在单线程测试中正确，但在并发环境下危险"的 bug。如果没有 code review，这个 bug 会在真实使用中导致消息错乱。

**writing-plans 帮助我建立了清晰的实现路线图。** 13 个 Phase、30+ 个 task，每个 task 有明确的文件路径和验证步骤。这个计划让我在实现阶段不会迷失方向——我知道下一步该做什么，做完后怎么验证。

## 哪些"形式大于实质"

**brainstorming 的某些追问过于模板化。** AI 会追问"你的目标用户是谁"、"为什么值得做"这类问题，但这些问题的答案在课程项目的语境下是显而易见的。对于"AI4SE 期末项目"这种场景，用户就是老师和助教，"值得做"是因为这是作业要求。把 brainstorming 当成"与智能体共同设计"的训练是有价值的，但 AI 有时会过度发散，提出一些在课程约束下根本不可行的建议（比如建议用 Redis 做会话存储）。

**executing-plans 和 subagent-driven-development 的边界有些模糊。** 对于我这种"一个人用一个 agent 连续工作"的场景，这两个技能的区别不大。我实际采用的是"手动逐 task 推进"的方式，而不是真的派多个 subagent 并行工作。课程要求"每个 task 由一个 subagent 完成"，但在实践中，我一个人连续与同一个 agent 对话，效率比频繁切换 agent 高得多。

## TDD 强制在 AI 协作下是阻碍还是放大器

**是放大器。** 在 AI 协作下，TDD 的价值被放大了。原因有三：

第一，AI 写测试比人快得多。我描述一个功能的预期行为，AI 能在几秒内生成完整的测试代码。这消除了"写测试太花时间"的借口。

第二，测试是 AI 实现代码的"护栏"。当 AI 写实现时，已有测试作为 spec 约束了它的行为。我多次看到 AI 的实现方案因为测试失败而被自动修正——如果没有测试，这些错误会进入代码库。

第三，测试是"人机协作"的沟通桥梁。当我对 AI 说"这个功能应该这样工作"，AI 先写测试，我确认测试表达了正确的意图，然后 AI 再实现。这个过程比"AI 直接写代码，我 review 代码"更高效，因为测试比代码更容易理解。

但有一个陷阱：AI 有时会写出"测试只测试自己"的测试——即测试和实现耦合在一起，改实现就改测试。我在 mock 测试中看到了这个倾向。解决方案是：要求 AI 在写测试时不能 reference 实现细节，只能基于 spec 描述行为。

## Subagent-driven 工作流让智能体能自主运行多久

在我的实践中，subagent 在**单个 task 粒度**（约 2-5 分钟）内表现最好。超过这个范围，agent 会开始偏离主题。例如，在 Phase 3 的"工具注册表 + 6 个工具"task 中，agent 在写完 3 个工具后开始"创新"——它想给 GitOps 加上 rebase 支持，虽然 PLAN 中没有这个需求。

**最优的 task 颗粒度**：一个 task 对应一个文件或一组紧密相关的文件，包含明确的"验证步骤"。比如"写 file_reader.go 和 file_writer.go，运行 `go test ./internal/tools/` 验证 3 个 registry 测试通过"——这个颗粒度下 agent 几乎不会偏离。

**最长的自主运行**：code review agent 自主运行了约 3 分钟，审查了 14 个文件的 3400 行代码，输出了 16 个 findings。这个过程中我没有干预，效果很好。这说明"审查"类任务（输入→分析→输出）比"创造"类任务（输入→设计→实现）更适合 agent 自主运行。

## SPEC / PLAN 质量如何影响实现质量

**规约不清导致 subagent 偏离的具体案例：**

在 Agent 主循环（Phase 6）的实现中，PLAN 中写了"反馈闭环：每个 turn 结束后只跑一次"。但我的实现中，反馈闭环在 `shell_run` 后也触发——因为 `codeModified` 的判断条件包含了 `action.Tool == "shell_run"`。Code review 的 agent 发现了这个问题，指出"对 `ls`、`echo` 这样的命令也触发 build 是浪费 CPU"。

这个问题的根因是 SPEC 中"触发条件"写得不精确。SPEC 说"写入 .go 文件后"，但 PLAN 中写成了"代码修改后"。如果 SPEC 从一开始就明确"只有 file_write 操作触发反馈闭环"，实现就不会出错。

**另一个案例**：上下文窗口管理的 token 估算。SPEC 说"使用 tiktoken 兼容算法"，但实现阶段用了简单的 4 chars/token heuristic。这是因为 tiktoken 的 Go 实现依赖复杂，我没时间集成。如果 SPEC 在技术选型时就考虑到实现成本，用一个"先简单 heuristic，后精确 tiktoken"的分阶段方案，就不会有这个 gap。

## 最有效的 prompt / context 策略

**"先写测试，再写实现"是我最有效的 prompt 策略。** 每次我说"先写这个测试，验证它失败，然后写实现"，agent 的产出质量明显高于"直接写代码"。测试固化了我的意图，agent 不会猜测。

**提供具体的文件路径和验证命令。** 当我给 agent 一个明确的文件路径（如"写 `internal/parser/parser.go`"）和验证命令（"运行 `go test ./internal/parser/ -v`，预期 4 个测试全部 PASS"），agent 几乎不需要追问。模糊的指令（如"写一个解析器"）则会导致反复沟通。

**CLAUDE.md 作为"持久化 context"非常有效。** 我把项目架构、当前进度、关键设计决策都写在 CLAUDE.md 中，每次新会话 agent 能快速理解上下文。这比每次重新解释项目背景节省了大量时间。

## 凭据与分发这两条工程要求

**凭据安全问题**迫使我想清楚了一个原本会忽略的链路：API Key 从哪来 → 怎么存 → 怎么用 → 怎么防止泄露。我设计了 ENV 读取 → MemoryStore → Provider 注入的流程，并在代码中实现了 MaskKey 脱敏。但我也意识到真正的 OS keychain 集成（go-keyring）在实际部署中是必要的——用户不应该手动管理环境变量。

**分发问题**让我重新审视了项目定位。Go 单文件二进制确实是"零依赖分发"的最简方案，但它也限定了一些设计决策：不能依赖外部数据库（所以用 SQLite 嵌入式）、不能依赖系统工具（所以 searcher 用 Go 原生实现而非系统 grep）。这些约束在早期设计时就需要考虑，否则后期会遇到"能编译但不能运行"的问题。

## 如果重做

1. **在 brainstorming 阶段就明确前端工具链。** 我在 Phase 11 才发现老师要求用 Open Design 插件，导致 Web UI 需要重做。如果一开始就确认前端技术栈，可以节省时间。

2. **PLAN 的 task 颗粒度要更细。** 特别是"工具注册表 + 6 个工具"这个 task，应该拆成 7 个独立 task。

3. **冷启动验证应该覆盖更多 task。** 我只验证了 2 个 task，但至少应该验证 3-4 个不同 phase 的 task 才能充分暴露 spec 缺陷。

4. **先做 CI 配置。** 我在实现全部完成后才意识到需要 CI，但 CI 应该在第一个 task 完成时就配置好——这样每次 push 都能自动验证，而不是最后一次性配置。

## 对 Superpowers 方法论的批判

**Superpowers 假设了什么：**
1. 用户有足够的上下文窗口来维持长期对话
2. 用户愿意在每个 task 后停下来做 review
3. 多个 subagent 可以并行工作
4. 用户了解每个技能的作用和触发条件

**这些假设在我的项目里成立吗：**

假设 1 **部分成立**。在实现阶段，连续 12 个 Phase 的对话保持了很好的连续性。但在 brainstorming 阶段，AI 的某些追问显得冗长，和当前 task 无关。

假设 2 **部分成立**。在早期 Phase 我确实每个 task 都停下来 review，但随着对 agent 的信任增加，我越来越倾向于让它连续推进多个 task。这既有效率提升（连续推进减少了 context 切换），也有风险（连续推进中积累的错误可能更晚发现）。

假设 3 **不成立**。对于单人项目，subagent 并行工作实际上效率更低——我需要同时管理多个 agent 的上下文，而它们的产出质量参差不齐。我最终采用了"一个 agent 顺序推进"的方式，效果更好。

假设 4 **不成立**。我花了不少时间理解每个 Superpowers 技能的作用和触发条件。比如我一开始不知道 `frontend-design` 和 `design-sync` 的区别，也不清楚什么时候该用 `executing-plans` 什么时候该用 `subagent-driven-development`。如果有一个"技能地图"（什么时候用什么技能），会大大降低学习成本。

**Superpowers 最有价值的部分**不是技能本身，而是它所倡导的**工程纪律**：先设计后编码、先测试后实现、先审查后合并。这些纪律在 AI 协作下没有被削弱，反而被强化了——因为 AI 让我们更容易"跳过"这些步骤（直接写代码更快），但跳过的代价是更高的 bug 率和更差的代码质量。

**Superpowers 最需要改进的部分**是技能之间的过渡。从 brainstorming 到 writing-plans 到 subagent-driven-development 到 code-review，每个阶段之间的切换需要人工判断时机。如果有一个"进度仪表盘"告诉我当前在哪个阶段、下一步该做什么、哪些 task 待完成，会让整个流程更顺畅。

**最终结论**：Superpowers 是一套有价值的工程方法论，但在单人项目中使用时，需要根据实际情况调整。并行 subagent 的假设对个人开发者不现实，但 TDD 强制、code review、冷启动验证这些核心实践确实提升了代码质量。我学到的最重要的一课是：**AI 可以加速编码，但不能替代工程纪律。**
